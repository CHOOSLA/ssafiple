---
title: Git 커밋 스타일 가이드
---

모든 Git 커밋은 아래 표준 스타일을 철저히 준수합니다.

## 커밋 메시지 포맷
```text
<Type>(<Scope>) : <한글 설명>
```
(주의: 닫는 괄호와 콜론 사이, 콜론 뒤에 공백이 위치하는 ` ) : ` 형태를 유지해야 합니다.)

## 예시
* `chore(git) : gitignore 설정`
* `feat(post) : 게시글 소프트 딜리트 API 구현`
* `fix(comment) : 댓글 조회 오류 수정`
* `docs(agents) : AI 지침 및 커밋 스타일 가이드 문서 업데이트`

## 커밋 타입 규칙
* `feat`: 신규 기능 추가
* `fix`: 버그 수정 및 버그 패치
* `docs`: 문서 생성 및 편집 (README, SCHEMA, AGENTS 등)
* `style`: 코드 스타일 수정, 포맷팅 (로직 변경 없음)
* `refactor`: 비즈니스 로직 리팩토링
* `chore`: 빌드, 가상환경, 패키지 설정 수정 등 (기능 구현 외 기타 개발 환경)

## 커밋 단위 원칙 (Commit Granularity)
브랜치를 나누지 않고 3인이 모두 `master`에서 직접 작업하므로([cooperation.md](cooperation.md) 참고), 커밋을 작고 자주 나누는 것이 충돌과 되돌리기 비용을 줄이는 유일한 안전장치입니다.

1. **도메인 경계 준수**: 한 커밋에 서로 다른 도메인(지도/게시판/AI·채팅) 파일을 섞지 않습니다. 담당 외 도메인 파일이 diff에 포함돼 있다면 커밋 전에 분리합니다.
2. **기능 단위 분리**: 하나의 커밋은 하나의 논리적 변경만 담습니다. 예: "게시글 생성 API 구현"과 "게시글 삭제 API 구현"은 완성 시점이 달라도 각각 별도 커밋으로 남깁니다. 서로 다른 기능을 몰아서 한 커밋에 담지 않습니다.
3. **관심사 분리 (Separation of Concerns)**:
   * 같은 기능이라도 계층이 다르면 나눌 수 있으면 나눕니다 (예: 백엔드 API 구현 커밋 → 프론트엔드 연동 커밋).
   * 모델/스키마 변경, 라우터·서비스 로직 변경, 스타일링 변경은 서로 다른 관심사이므로 가능한 한 분리합니다.
   * `chore`(환경설정)·`refactor`(동작 변경 없는 구조 개선)·`style`(포맷팅)은 `feat`/`fix`와 절대 섞지 않습니다.
4. **지속적 커밋 (Commit Early, Commit Often)**: 작업을 끝까지 몰아서 한 번에 커밋하지 않고, 아래와 같은 자연스러운 체크포인트마다 커밋합니다.
   * 함수·엔드포인트·컴포넌트 하나가 동작 가능한 상태에 도달했을 때
   * 다른 파일/작업으로 전환하기 직전
   * `git pull origin master`로 최신화하기 직전 (충돌을 작은 단위로만 만나기 위해)
   * 되돌릴 가능성이 있는 실험적 변경을 시도하기 직전
5. **커밋 크기 가이드**: 파일 5개 이상 또는 서로 다른 라우터/뷰를 동시에 건드리는 커밋은 분리를 검토합니다. 반대로 오타 수정 같은 사소한 변경까지 지나치게 잘게 쪼갤 필요는 없습니다.
