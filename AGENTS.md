## 4. 데이터베이스 및 시딩(Seeding) 제약사항

### 4.1. Soft Delete 기반 연쇄(CASCADE) 처리
* 물리적 삭제(`DELETE`)를 배제하고, `is_deleted = True` 플래그를 업데이트하는 **소프트 딜리트(Soft Delete)** 방식을 적용합니다.
* **게시글이 Soft Delete 처리될 경우, 해당 게시글에 종속된 모든 댓글도 연쇄(CASCADE)적으로 `is_deleted = True` 처리**되도록 백엔드 비즈니스 로직에서 구현해야 합니다.
* 모든 데이터 조회(목록/상세) 시 `is_deleted = False`인 데이터만 필터링하여 노출합니다.

### 4.2. 위도/경도(좌표) 뒤바뀜 방지 검증 (Assert)
* 공공데이터 JSON의 지리 정보(`mapx`, `mapy`) 적재 시, 위도와 경도가 뒤바뀌는 사고가 빈번합니다.
* `seed.py`에서 데이터를 파싱 및 적재할 때, **서울 권역 경계선(Bounding Box) 범위 검증을 강제(Assert)로 수행**합니다.
  * **경도 (Longitude)**: `126.75 <= longitude <= 127.20`
  * **위도 (Latitude)**: `37.40 <= latitude <= 37.72`

## 8. Git 커밋 스타일 가이드
프로젝트 진행 시 생성되는 모든 Git 커밋은 사용자가 지정한 표준 스타일 규격을 철저히 준수합니다.

* **커밋 메시지 포맷**:
  ```text
  <Type>(<Scope>) : <한글 설명>
  ```
  *(주의: 닫는 괄호와 콜론 사이, 콜론 뒤에 공백이 위치하는 ` ) : ` 형태를 유지해야 합니다.)*

* **타입(Type) 및 범위(Scope) 예시**:
  * `chore(git) : gitignore 설정` (표준 예시)
  * `feat(post) : 게시글 소프트 딜리트 API 구현`
  * `fix(comment) : 댓글 조회 오류 수정`
  * `docs(agents) : AI 지침 및 커밋 스타일 가이드 문서 업데이트`

* **커밋 타입 규칙**:
  * `feat`: 신규 기능 추가
  * `fix`: 버그 수정 및 버그 패치
  * `docs`: 문서 생성 및 편집 (README, SCHEMA, AGENTS 등)
  * `style`: 코드 스타일 수정, 포맷팅 (로직 변경 없음)
  * `refactor`: 비즈니스 로직 리팩토링
  * `chore`: 빌드, 가상환경, 패키지 설정 수정 등 (기능 구현 외 기타 개발 환경)
